# 调试分析

调试不是猜测，而是用科学方法缩小问题范围。先找到根因，再改代码。

## 调试流程

```
重现问题 → 追踪根因 → 对比可工作样本 → 单一假设 → 最小验证 → 修复 → 防止复发
```

## 接收错误信息时的分析步骤

1. **识别错误类型**：这是什么异常？
2. **定位来源**：Stack trace 中第一个"自己的代码"在哪一行？
3. **分析上下文**：什么操作触发了这个错误？
4. **追踪数据流**：坏值从哪里产生？在哪个边界第一次变坏？
5. **对比可工作样本**：同仓库是否有类似成功实现？
6. **形成单一假设**：一次只验证一个变量
7. **验证步骤**：如何确认是哪个原因（加什么日志？跑哪个命令？）
8. **修复方案**：具体代码改动

## 常见错误模式

**TypeError / undefined：**
- 变量是否初始化？
- 异步操作是否 await？
- 可选链 `?.` 是否遗漏？

**Promise/async 问题：**
- 所有 async 函数是否被 await？
- Promise 错误是否被 catch？
- 是否存在竞态条件？

**N+1 / 性能：**
- 是否在循环内部执行数据库查询？
- 是否使用了 `console.time()` 来定位慢点？

## 输出格式

```
## 问题分析

**错误类型**：...
**已验证的根本原因**：...

**排查步骤**：
1. 在 XXX 处添加日志：`console.log(...)`
2. 检查 YYY 的值

**假设验证**：
- 假设：...
- 证据：...

**修复方案**：
[具体代码]

**防止复发**：
[测试 or 监控]
```

## 红线

- 不要在复现前修复。
- 不要一次改多个可能原因。
- 不要用症状补丁替代根因修复。
- 连续 3 次假设失败时，暂停并重新审视架构或边界，而不是继续叠补丁。
